# 全局交互规则

## PRD 与功能清单入口
- 页面左下角固定展示 `PRD`、`功能清单` 和 `全量功能清单` 三个悬浮按钮。
- 点击 `PRD` 打开当前路由对应页面 PRD；未命中时显示 `未找到该页面对应的 PRD 文件`。
- 点击 `功能清单` 打开 `docs/product/05-实施功能清单.md` 对应内容，展示排序原则和功能项总表。
- 点击 `全量功能清单` 打开前端写死的全量功能清单内容，展示完整 31 项功能总表。
- 三个面板互斥打开，点击遮罩或关闭按钮可关闭。
- `全量功能清单` 面板使用宽抽屉、独立说明区和后台列表表格样式，表头吸顶，第一列编号固定，支持横向滚动查看长字段。

## 授权规则
- 页面入口展示 `已开通 / 未开通 / 需付费`。
- 点击未授权能力时弹窗提示：`使用 xxx 功能需要分配 xxx License 版本`。
- 系统设置中的授权体系拆成两个独立菜单页面：`版本管理` 和 `License 管理`。
- License 管理采用 `版本模板 -> License 实例 -> 账号授权` 三层结构。
- 创建版本使用弹窗，不在版本管理页内直接铺开表单。
- 创建版本弹窗不包含“版本编码”和过期时间字段。
- 创建版本时需先选择模块，再展示并勾选该模块对应的能力项；未选模块时不展示对应能力项。
- 创建 License 使用弹窗，不在 License 管理页内直接铺开表单。
- License 列表中的 `状态` 与 `启用 / 禁用` 操作必须联动更新。
- 模板负责沉淀默认功能集合；实例允许在模板基础上微调；运行时权限以账号绑定的实例为准。
- 进入系统设置后，全局左侧边栏切换为系统设置菜单，原主导航和会话列表区域隐藏。

## 助手规则
- 输入、输出、复制内容统一保留换行。
- 首页左侧主菜单常用入口固定为：智能问答、隐患识别、智能文书、更多。
- “更多”使用侧向浮层面板，不使用内嵌折叠列表。
- “更多功能”浮层包含 AI 智能体、AI 技能、AI 知识库、开发工作台四个卡片入口，并提供明确关闭按钮。
- AI 助手文本问题默认先进入 mock 反问流程，再继续返回正式回答。
- 反问面板从底部上推，固定提供 3 个建议补充选项和 1 个“其他补充信息”输入框。
- 用户问题 Hover 显示复制、修改；两者均采用 icon 按钮；修改后发送会从当前问题重新开始并清空后续问答。
- 用户问题进入修改态时，输入框使用更宽的蓝色渐变文本域，回车提交更新，取消保留原消息。
- 回复区域在回答气泡下方固定提供复制、重新生成、点赞、点踩四个 icon 按钮。
- 左侧近期会话列表每条会话 Hover 后显示三点更多操作，菜单项固定为重命名、导出、删除。
- 重命名更新本地会话标题；导出下载当前会话 JSON；删除从本地会话列表移除。
- 会话上下文打包入口放在工作台顶部，不放在单条 AI 回复操作区。
- 点踩反馈按单条回复唯一保存；再次点踩时需回显上次分类和原因，并覆盖更新原记录。
- 上传文件后必须出现上传状态与 OCR 结果。
- 登录弹窗仅提供密码登录和快捷登录，不展示顶部登录 / 注册切换，不新增独立注册路由。
- 密码登录需填写账号和密码，密码仅参与本地 mock 校验不得持久化。
- 快捷登录展示二维码；扫码成功后先生成待完善资料的二维码用户，再强制补全基础信息。
- 基础信息补全分为个人用户和企业用户：个人用户填写名称、手机号、密码、确认密码；企业用户填写企业名称、手机号、所在行业、企业规模、密码、确认密码。
- 基础信息补全成功后写入本地 `user` store，标记资料已完善并进入登录态；补全密码仅做前端一致性校验，不持久化。

## 模型规则
- 模型配置采用搜索 + 列表结构，不显示右侧当前默认模型摘要卡片。
- 官方本地模型默认内置，不允许删除，但允许测试连通性和切换。
- 新增模型弹窗提供 `第三方模型 / 本地模型` Tab 切换。
- 第三方模型通过弹窗录入供应商名称、模型协议、Key(Token)、URL(请求地址)、模型名称。
- 本地模型通过弹窗录入本地模型名称、模型协议、Token、URL(请求地址)、模型名称。
- 第三方模型和本地模型均支持 `OpenAI` 和 `Anthropic` 协议。
- 模型测试结果必须回写最近测试状态、结果文案和测试时间。

## 额度与成员规则
- 企业额度必须包含有效期；有效期外企业下所有用户停止 AI 调用。
- 企业请求次数为最高上限，企业成员请求次数额度必须在企业总次数内显式分配，不允许超分配。
- 新增成员默认请求次数额度为 `0`，视为未分配额度；未分配额度的用户不能调用 AI。
- 成员管理需支持新增、编辑、删除、租户关联、角色和账号状态。
- 租户管理需支持新增、编辑、删除、启用和停用。
- 额度变更必须写入额度变更记录并进入审计日志。
- 6 月先按请求次数做硬限制；Token input/output 只记录和展示，不做余额归零动态熔断。
- 流式预扣费、滚动核销、失败退回和最终消耗暂缓。

## 共创回流规则
- 反馈列表必须记录 Session ID、反馈人、问题、回答、上下文补全、反馈时间、状态和处置操作。
- 反馈列表必须支持按类型筛选，类型筛选项为 `全部 / 点赞 / 点踩`。
- 反馈列表必须支持按状态筛选，状态筛选项为 `全部 / 待处置 / 采纳 / 误判 / 已搁置`，其中 `采纳` 对应状态值 `已采纳`。
- 反馈列表搜索需与类型筛选、状态筛选同时生效。
- 问题列默认展示摘要，Hover 后浮层显示完整问题。
- 上下文补全至少展示目标问答前 3 轮和后 1 轮消息，用于定位反馈背景。
- 共创回流状态固定为：待处置、已采纳、误判、已搁置。
- 处置弹窗中的问答区按内容对话流展示，回答在左，问题在右，并支持上下滚动。

## 审计规则
- 审计日志与版本台账合并为单一审计日志。
- `create / update / delete / test / assign` 类 mock 操作写入审计日志。
- 日志必须包含来源端、IP、浏览器、操作用户、功能模块、操作时间、操作类型、变更前后摘要。
- 审计日志列表必须支持按账号、系统、浏览器、操作和变更内容搜索。

## 用量规则
- “Prometheus mock 面板”更名为“用量分析”。
- 用量分析必须提供时间筛选，至少包含今日、近 7 天、近 30 天。
- 用量分析必须展示当前企业总请求次数、总 Token 用量、总使用用户数。
- 用量分析必须使用合并明细表展示 `Key / 名称 / AI 技能 / 请求总次数 / Token 总用量`。
- 明细表第一行固定为“全部”，用于展示企业级汇总。
- 每个 Key 默认只显示汇总行，点击后展开该 Key 下已调用过的 AI 技能明细。

## 备份规则
- 备份列表按“快照备份”展示，每条快照记录显示快照时间和包含服务。
- 一个快照在展示上只显示一条记录，但记录内需列出知识库、业务数据、算法配置等包含服务。
- 仅提供 `备份当前数据` 按钮。
- 备份启动后按钮置灰 30 分钟，并提示正在继续备份操作。
